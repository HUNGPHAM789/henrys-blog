Inside ARIA Autopilot: The Architecture of a Self-Driving AI Agent

2026-04-14
← Back to blog

Yesterday I wrote about teaching a 26B model to finish its tasks. Today I want to go deeper — into the actual architecture of the task runner system that makes it work.

The core idea: the model is the worker, the code is the foreman. The model never decides when to start, stop, or what to do next. An external orchestrator drives every step.

The Router: Four Paths for Four Problems

Every message from the user hits a classifier first. It's regex-based — no LLM call, instant.

classifyTask(message) → chat | simple-task | complex-task | debug-task
  • Chat — "what time is it?", "explain X" → straight to the model, no orchestration
  • Simple task — "write a script to do X" → plan + execute, no approval needed
  • Complex task — "write a blog post and publish it" → full plan mode with exploration and approval
  • Debug task — "the blog post is broken" → diagnose-fix-verify loop

The classifier looks at message length, keyword patterns, and complexity signals. A 20-character message is always chat. "Fix" or "broken" routes to debug. Multi-domain requests with sequential language ("and then", "first...then") route to complex.

Path 1: Chat — Direct Model Call

Simple questions go through runClaude() — the core agentic loop. Despite the name (legacy from the Claude SDK days), it talks to Ollama.

The loop runs up to 40 turns:

  1. Send message + conversation history (last 20 from DB) to Ollama
  2. Model responds — tool calls? Execute them, feed results back, loop
  3. No tool calls? Check for continuation signals. If model says "now I'll do step 2" but didn't call a tool → inject "Continue. Use tools." (max 3 nudges)
  4. If model went silent after a tool call (empty response) → force one more turn: "Summarize what you did"

Three safety nets stacked: continuation detection catches the "I'll do it next" stall. Silent-exit forces a summary. And the 3-nudge cap prevents infinite loops.

Path 2: Simple Task Runner — Plan + Execute

For straightforward tasks, the runner takes over:

  1. Plan: Ask the model to break the task into 2-6 numbered steps
  2. Validate: Reorder if deploy comes before create. Deduplicate similar steps.
  3. Execute each step: Feed individually — "Do step N now. Use tools." Each step gets a 2-minute timeout via Promise.race().
  4. Judge each step: Regex checks for success ("done", "created", "✅") or failure ("error", "blocked"). If unclear but tools were used, call it done. If failed, retry once.
  5. Summarize: Force a final message starting with ✅ or ❌.

Key detail: each step gets the full result of previous steps (up to 2KB) as context. Not just "Step 1 done" — the actual file contents, command outputs, whatever the model produced. This means Step 3 can reference what Step 1 read without re-reading it.

Path 3: Plan Mode — Explore, Design, Approve, Execute

This is the big upgrade. For complex tasks, the runner adds two phases before execution:

Phase 1: Explore

The model gets tools and a clear instruction: "Before planning, investigate. Read files, check git status, search memory. Report what exists." This is 12 turns of pure investigation — no writing, no building.

This is why the blog post came out wrong before. Without exploration, the model guessed at the HTML structure. With exploration, it reads the actual template, existing posts, and index.html before deciding anything.

Phase 2: Design

Feed the exploration findings to the model: "Based on what you found, design the approach. What are the risks? What's the sequence?" This produces a specific plan with actual file paths, not generic steps.

Phase 3: Approve

The plan goes to the user on Telegram with inline buttons:

[✅ Approve] [✏️ Modify] [❌ Cancel]

Nothing executes until the user taps Approve. If they tap Modify, the system asks what to change and replans. Cancel drops everything.

Phase 4: Execute

Same step-by-step loop as the Simple Task Runner, but with the exploration context injected. Every step knows what the model discovered in Phase 1.

Path 4: Debug Runner — Diagnose, Fix, Verify, Loop

When something is broken, the pattern is fundamentally different from building something new:

  1. Diagnose: "Boss reported [issue]. Investigate with tools. What's wrong?" (10 turns)
  2. Fix: "Based on diagnosis, fix it NOW. Use tools." (15 turns)
  3. Verify: "Test if the fix works. Curl the URL, read the file. Answer YES or NO." (8 turns)

If verification fails — loop back. Up to 3 cycles. Each cycle gets the context from previous attempts, so it doesn't repeat the same fix twice.

Background Agents: Same Architecture, Async Execution

When the user asks ARIA to research something or build something big, ARIA can spawn background agents. These are isolated runTask() calls running asynchronously.

The agent goes through the same router. A researcher agent's prompt gets classified, routed to the appropriate runner, and executed with the same plan-execute-verify pattern. ARIA's main session stays responsive while agents work in the background.

Completion notification comes through a 15-second poller that checks the database for finished tasks.

The Foundation: runClaude()

Every path — chat, simple, complex, debug, background agents — funnels through one function at the bottom: runClaude(). The runners don't do inference. They compose multiple runClaude() calls with different prompts, each scoped to one job.

  • Explorer phase: runClaude("investigate", maxTurns=12)
  • Design phase: runClaude("plan the approach", maxTurns=3)
  • Step execution: runClaude("do step N", maxTurns=15)
  • Verification: runClaude("test if it works", maxTurns=8)
  • Summary: runClaude("summarize for Boss", maxTurns=3)

The runner is a state machine. The model is a function it calls.

What This Means for Local AI

A 26B model can't reliably manage a 5-step task on its own. But it can reliably execute one step when told exactly what to do. The Autopilot architecture exploits this — decompose, isolate, verify, repeat.

This pattern works with any model that supports tool calling. Swap Gemma 4 for Llama, Qwen, or Mistral — the architecture doesn't change. The model is a replaceable component. The orchestration is the product.

🔑The lesson: a dumb model in a smart system outperforms a smart model in no system. Structure beats intelligence when reliability matters.
