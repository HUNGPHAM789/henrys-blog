ARIA Autopilot: How I Taught a 26B Local AI to Actually Finish Its Tasks

2026-04-13
← Back to blog

I spent three days asking my local AI assistant "are you done?" and getting silence. Here's how I fixed it — no cloud APIs, no fine-tuning, just code.

The Problem: Silent AI

I run a local AI assistant called JarvisM4 on my Mac Studio M4 Max. It uses Gemma 4 26B through Ollama — fully local, no API costs, 66 tokens/sec. For simple questions, it's great. But ask it to do a multi-step task — "write a blog post and publish it" — and it breaks down.

The pattern was always the same:

  • I send a task via Telegram
  • The AI reads a file (Step 1)
  • It says "Great, now I'll do Step 2..."
  • Silence. Forever.

The model would call a tool, get the result, and treat that as the end of its turn. No follow-up. No summary. No "here's what I did." I'd check back 30 minutes later and realize nothing happened after Step 1.

💡This isn't a Gemma-specific bug. Any local model under ~30B parameters will occasionally "forget" multi-step instructions. The model's attention drifts — it focuses on the immediate tool result and loses the bigger picture.

What Didn't Work

Prompt engineering. I added rules to the system prompt: "ALWAYS report back after every tool call." "NEVER go silent." "Your FINAL message MUST be ✅ or ❌." The model followed these rules... sometimes. Maybe 60% of the time. Not enough.

OpenClaw (the previous system). I tried OpenClaw, a full-featured AI gateway. It has heartbeats, hooks, a dashboard, cron jobs. But its heartbeat kept firing overnight into a broken session, bloating the context to 96K tokens. The model choked on the history and went completely unresponsive. I spent a full day debugging pairing loops, session bloat, and config schema bugs.

The Solution: ARIA Autopilot

I stopped trusting the model to manage itself and built an external orchestrator. The model is the worker. The code is the boss.

1. Task Router

Every incoming message gets classified:

  • Chat — simple question, goes straight to the model
  • New task — "write X", "build Y", "deploy Z" → Task Runner
  • Debug task — "it's broken", "fix X", "not working" → Debug Runner

2. Task Runner (Plan → Execute → Verify)

For new tasks, the runner takes control:

  • Plan: Ask the model to break the task into 2-6 numbered steps
  • Execute: Feed each step individually — "Do step N now. Use tools."
  • Judge: After each step, check: did it actually use tools? Did it work?
  • Retry: If a step fails, retry up to 2x before moving on
  • Summarize: Force a final summary starting with ✅ or ❌

The model never decides when to stop. The runner does.

3. Debug Runner (Diagnose → Fix → Verify → Loop)

For bug reports, the runner uses a different pattern:

  • Diagnose: "Investigate the problem. Use tools to check current state."
  • Fix: "Based on the diagnosis, fix it NOW."
  • Verify: "Test if the fix works — curl the URL, read the file."
  • If still broken → loop back (up to 3 cycles)

4. Continuation Detection

The biggest fix was catching the "Step 1 done, now Step 2..." exit pattern. When the model responds with text mentioning future steps but makes no tool calls, the loop injects: "Continue. Do the next step now — use your tools." and keeps going.

5. Silent Exit Safety Net

If the agentic loop ends with a tool result as the last message (meaning the model went silent), it forces one more inference: "You just used a tool but didn't reply to Boss. Summarize what you did."

The Result

Before ARIA Autopilot: I'd send a task and check back 5 times asking "are you done?"

After: The system drives the model through every step and guarantees a ✅ or ❌ at the end. The model can still think, plan, and use tools creatively — but the code ensures it actually finishes.

Other Fixes Along the Way

  • Conversation memory: ARIA wasn't saving messages to the DB. Every turn was a blank slate. Fixed by generating session IDs from thread IDs and loading the last 20 messages on each call.
  • Gemma 4 thinking mode: The model puts reasoning in message.thinking instead of message.content. If content is empty but thinking exists, we use thinking as the response.
  • Ollama timeouts: No timeout on API calls meant a hung model = hung bot. Added 5-minute AbortSignal on every request.
  • Skill templates: Telling the model "match the existing style" doesn't work. Giving it a 29KB HTML template file to copy from does.

Stack

  • Hardware: Mac Studio M4 Max, 36GB unified memory
  • Model: Gemma 4 26B via Ollama (Q4_K_M, ~16GB, 66 tok/s)
  • Framework: Node.js + TypeScript, Telegraf for Telegram
  • Database: SQLite + sqlite-vec for embeddings
  • Embeddings: Ollama nomic-embed-text (768-dim, local)
  • Repo: github.com/HUNGPHAM789/aria-mac
🔑The lesson: don't prompt-engineer reliability into a small model. Build it into the system around the model. The model is the worker, the code is the foreman.
