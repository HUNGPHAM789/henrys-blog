Agent Self-Improvement: How AI Systems Learn to Upgrade Themselves

Andrej Karpathy's autoresearch showed that AI can optimize AI — 11% efficiency gains in days, zero human intervention. Jarvis already does a version of this.

Henry's Blog · March 24, 2026 · 9 min read

In March 2026, Andrej Karpathy (the researcher who trained GPT-2 and co-founded OpenAI) open-sourced a project that sent the AI community into orbit: autoresearch. The idea: give an AI agent a codebase and a metric to optimize, let it run experiments automatically, and watch it find improvements humans hadn't discovered.

The results were startling. The system found 20–29 effective optimizations in days. Training efficiency improved by 11%. The agent wasn't just running code — it was conducting machine learning research autonomously.

But here's what most people missed: Jarvis already does a primitive version of this. The OpenClaw self-evolution protocol has been running since March 2026. This post explains the field, what Jarvis does, and how to take it further.

The Hierarchy of Self-Improvement

AI self-improvement exists on a spectrum from simple to profound:

  • Level 1 — Prompt optimization: The agent improves its own system prompts based on which ones produce better outputs. Already happening with OpenClaw's AutoResearch sandbox.
  • Level 2 — Tool configuration: The agent adjusts which tools it uses and how. Jarvis adds new MCP servers, updates TOOLS.md, improves AGENTS.md.
  • Level 3 — Code evolution: The agent modifies its own codebase to add capabilities. SICA (Self-Improving Coding Agent) demonstrates this — it edits its own code and runs benchmarks to measure improvement.
  • Level 4 — Training optimization: Karpathy's autoresearch territory — the agent modifies model training code to improve model performance. This requires GPU access and is beyond personal assistant territory for now.

For Jarvis and OpenClaw users, Levels 1 and 2 are immediately practical. Level 3 is the near-term frontier.

Karpathy's AutoResearch: The Breakthrough

GitHub: The concept was demonstrated in multiple open-source projects; the community collected resources at github.com/alvinunreal/awesome-autoresearch

The key insight Karpathy articulated is what he called the Verifiable Domain Thesis: AI systems can recursively self-improve in domains where outcomes are unambiguously measurable. Code has this property — you run the code, the tests pass or fail, the benchmark score goes up or down. There's no ambiguity.

The autoresearch loop works like this:

  1. Agent reads the current codebase and understands what metric to optimize
  2. Agent generates a hypothesis: "What if I change X to Y?"
  3. Agent implements the change and runs an evaluation
  4. If the metric improves, the change is kept. If not, it's reverted.
  5. Agent documents what worked and what didn't, and generates the next hypothesis
  6. Loop repeats hundreds of times until diminishing returns

This is exactly how human researchers work — except the AI runs experiments in minutes instead of weeks.

EvoAgentX — The Framework for Self-Evolving Agents

GitHub: github.com/EvoAgentX/EvoAgentX · Stars: 2,000+

EvoAgentX is an open-source framework purpose-built for building, evaluating, and evolving LLM-based agents. It has three standout features:

  • Automated workflow construction: Given a task description, EvoAgentX builds the agent workflow automatically — no manual configuration.
  • Integrated evaluation: Built-in evaluation harnesses that measure agent performance on defined metrics.
  • Self-evolution engine: Iteratively improve the agent's prompts, tools, and workflow based on evaluation results.

For OpenClaw: EvoAgentX provides the infrastructure to turn Jarvis's current manual self-evolution protocol (detect patterns → propose upgrades → Boss approves → apply) into a semi-automated system where Jarvis proposes and tests improvements independently before presenting the best ones for approval.

Gödel Agent — Real-Time Self-Modification

Published at: ACL 2025 · GitHub: Available in the EvoAgentX ecosystem

The Gödel Agent (named after mathematician Kurt Gödel, famous for incompleteness theorems) takes self-improvement to an extreme: it modifies its own task-solving policies and learning algorithms at runtime, using a technique called monkey patching — dynamically replacing Python functions while the agent is running.

The philosophical implications are interesting: an agent that can rewrite its own decision-making process is qualitatively different from one that just updates its prompts. It's closer to how evolution works — not just learning within fixed architecture, but changing the architecture itself.

For practical OpenClaw use: Gödel Agent is research territory, not production-ready. But understanding the concept helps design Jarvis's evolution system with the right architecture — one where reasoning strategies can be swapped out, not just parameters tuned.

What Jarvis's Self-Evolution Looks Like Right Now

The OpenClaw self-evolution protocol is real and active. Here's what it has already produced:

  • AGENTS.md upgrades: 15+ documented improvements over 2 weeks — parallel subagent rules, Plan-Mode gate, credential hygiene standards, gotchas documentation
  • AutoResearch sandbox: The autoresearch-openclaw/ directory runs Karpathy-inspired optimization loops on Jarvis's .md files using Gemini free tier — 74.1 → 77.0 score, saved 3,767 bytes
  • Coding patterns: Sibling component rules (fix all variants when fixing one), route verification (grep before pushing), large file generation (split into phases)
  • Tool stack evolution: Aider watch mode, Gemini CLI integration, Magic MCP for UI — all discovered and integrated through observed patterns

This is already impressive for a system that's been running for less than 2 weeks. The gap between current Jarvis and a fully self-improving system is mostly infrastructure: automated scoring, sandbox evaluation, and a better proposal/approval workflow.

The Next Evolution: Semi-Automated Improvement

The current protocol is: Jarvis observes a pattern → writes a "📝 Noted:" in daily logs → at end of session proposes upgrades → Boss approves → Jarvis applies. This works but requires Boss to be in the loop for every change.

A better system would:

  1. Let Jarvis propose and test improvements autonomously in the sandbox
  2. Present Boss with a weekly "evolution report" — what was tested, what improved, what didn't
  3. Boss approves the winning changes in bulk, not one at a time
  4. Critical system changes (SOUL.md, security rules) still require explicit approval

Boss Henry's role in self-evolution: Not technical. The evolution protocol should surface improvements as clear, plain-language proposals: "I noticed I was failing to check sibling components 3 times this week. I added a rule to AGENTS.md that requires grep verification before any component fix. Here's the diff — approve?" One tap to approve, one tap to reject. That's it.

✅ YES — Already Doing This

Verdict: Self-evolution is Jarvis's most important long-term capability — invest in it

The OpenClaw evolution system is working but manual. The next step is semi-automation using the EvoAgentX framework for structured evaluation and the autoresearch sandbox for testing changes before applying them.

Immediate action: Build a weekly "evolution report" script that summarizes what Jarvis learned, what patterns it detected, and what it wants to change. Present to Boss as a single approval session, not ad-hoc updates.

Longer term: Integrate EvoAgentX evaluation harnesses to give self-improvement proposals a score, not just a description. Turns "I think this is better" into "this improved task completion rate by 12%."
