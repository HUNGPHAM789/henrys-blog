Browser Automation + AI: Give Your Agent Eyes and Hands on the Web

The moment your AI agent can navigate a real browser, an entire category of workflows that were impossible become trivial.

Henry's Blog · March 24, 2026 · 8 min read

There are two kinds of AI agent tasks: things where an API exists, and things where you have to use a real browser. The first category is well-served — Supabase has an API, GitHub has an API, Google Calendar has an API. But what about pulling data from a Vietnamese government education website? Or filling out a form that predates the API era? Or checking if your Antera deployment looks right on mobile?

That's where browser automation comes in. In 2024–2025, the combination of AI + browser automation went from experimental to production-ready. The tools are good now. Let's cover the ones that matter.

Three Approaches to AI Browser Automation

Not all browser automation is the same. There are three distinct approaches, and understanding the difference saves you from picking the wrong tool:

Approach 1: Scripted Automation (Playwright + AI code generation)

You use AI to write Playwright scripts, then run them deterministically. This is the most reliable approach — once the script is written and working, it works every time. Playwright is Microsoft's modern browser testing library that works with Chromium, Firefox, and WebKit.

Best for: recurring workflows where the page structure doesn't change. Upload Zoom recordings, fill standard forms, check that Antera renders correctly on different screen sizes.

Approach 2: MCP Browser Control (browser-tools MCP)

The browser-tools MCP server connects your AI agent to a live Chrome browser via Chrome DevTools Protocol. Your agent can navigate pages, click elements, take screenshots, and read the DOM — all through MCP tool calls, in real time, as part of a larger workflow.

Best for: ad-hoc tasks where you want Jarvis to "go check X" during a session. Visual audits, form submissions, reading pages that block scrapers.

Approach 3: Autonomous Browser Agents (browser-use / Playwright AI agents)

The AI agent takes natural language instructions and controls a browser autonomously using computer vision and DOM analysis. You say "go to this site, find the login form, enter these credentials, navigate to settings, download the export." The agent figures out how to do it without a pre-written script.

Best for: one-off tasks on unfamiliar sites, research workflows, sites that change their structure frequently.

The Tools in Detail

browser-use — The Autonomous Agent Pioneer

GitHub: github.com/browser-use/browser-use · Stars: 78,000+

browser-use hit 78,000 GitHub stars because it solved the hard problem: making websites navigable by AI without pre-written scripts. It uses a combination of DOM analysis and computer vision to understand page structure, then uses an LLM to decide what to click, type, or navigate.

The experience is genuinely impressive. You give it a task in plain English — "log into my account on site X and download the monthly report" — and it figures out how to do it. It handles pagination, modals, CAPTCHAs (mostly), and dynamic JavaScript content.

For Boss Henry: this is the tool that could automate the Zoom recording download and categorization workflow entirely. Log in to Zoom, navigate to recordings, identify recordings by class name, trigger downloads — all from a single Jarvis command.

Playwright MCP Server

GitHub: github.com/microsoft/playwright-mcp · Stars: 12,000+

Microsoft's official MCP server for Playwright. This is the right choice when you want deterministic, scriptable browser control through MCP — not autonomous AI navigation, but precise programmatic control that your agent can call like any other tool.

The Playwright MCP exposes tools like navigate, click, fill, screenshot, get_text. Your Claude Code or OpenCode agent can call these directly during a task, without spawning a separate browser automation process.

Already available for the browser-tools MCP on the Jarvis machine. The Playwright MCP is the upgrade — cleaner API, better error handling, and direct integration with Claude Code's MCP client.

Firecrawl — The Data Extraction Layer

GitHub: github.com/mendableai/firecrawl · Stars: 82,000+

Firecrawl isn't a browser automation tool — it's a web data extraction service. The difference matters: Firecrawl turns any webpage into clean, structured markdown or JSON that AI can work with. It handles JavaScript rendering, pagination, and anti-scraping measures.

Use Firecrawl when you want to read web content, not interact with it. Research, content extraction, competitor analysis, monitoring web pages for changes.

Firecrawl has a generous free tier and an MCP server. It pairs perfectly with browser-use: Firecrawl reads the data, browser-use takes the actions.

The ESL Teacher Workflow That Browser Automation Unlocks

Boss Henry's monthly teaching workflow is currently 7 manual steps. Here's what AI + browser automation could reduce it to:

1

Jarvis triggers browser-use to log into Zoom

Natural language instruction: "Log into Zoom and navigate to the recordings page for this month." browser-use handles authentication and navigation.

2

Playwright MCP downloads recordings

Deterministic script: click each recording matching class name pattern, trigger download, save to designated folder.

3

Jarvis reads zoom-report.js output

The existing zoom.js script counts sessions per class. Jarvis combines this with the downloaded recordings list.

4

Jarvis generates report and notifies Boss via Telegram

The report arrives as a formatted message. Boss reviews, approves, and it's sent to Facebook groups. Total manual time: under 2 minutes.

Current status: The browser-tools MCP is installed on the Jarvis machine. The Zoom API scripts exist. The missing piece is browser-use for the authentication step — Zoom's OAuth flow doesn't work cleanly with pure API calls, but browser-use can handle it. This is a high-value workflow automation sitting one installation away.

The Limitations to Know Before You Start

CAPTCHAs are the enemy. browser-use can handle simple CAPTCHAs but complex ones (reCAPTCHA v3, hCaptcha) will block autonomous agents. Sites that actively resist automation need workarounds.

Dynamic sites break scripted approaches. If a site redesigns its UI, all your Playwright scripts need updating. The autonomous approach (browser-use) handles this better but is slower and more expensive.

Browser sessions are resource-intensive. Running a real Chrome browser uses 200–500MB RAM. For lightweight extraction, Firecrawl's headless approach is much more efficient.
