Long-Task Tracking and Agent Notification Systems

You shouldn't have to ask "are you done?" — a well-designed agent tells you the moment it finishes, and never leaves work dangling without a status update.

Henry's Blog · March 24, 2026 · 8 min read

There's a specific hell that comes with autonomous AI agents: you send them off on a task, and then you have no idea what they're doing. Is it stuck? Did it fail silently? Did it finish and not tell you? Are you waiting for nothing?

This is the notification problem. For simple synchronous tools — press button, get result — it doesn't matter. But for AI agents that take 5–30 minutes on complex tasks, poor notification design means Boss Henry is either constantly checking in (losing the value of delegation) or missing completed work (creating invisible bottlenecks).

This post covers the patterns and tools that solve long-task visibility for AI agent systems like OpenClaw.

The OpenClaw Notification System

Jarvis already has a working notification system built around Telegram and notify-boss.js. Here's what exists:

  • notify-boss.js: Sends a Telegram message directly to Boss Henry. Used at the end of long tasks.
  • run-claude.js: Wires Claude Code subagents to automatically notify on completion (success or failure) with elapsed time.
  • Mission Control dashboard: localhost:3333 shows active agents, scheduled jobs, project status, and event log.
  • Heartbeat system: Periodic polling of HEARTBEAT.md to check for pending tasks and push updates.
  • active-tasks.json: Handoff file for tasks that are waiting or paused while Boss is AFK.

This is already good. But there are gaps: subagent failures sometimes go unnoticed, task queues can pile up without Boss knowing, and there's no structured SLA for "task should complete in X minutes — alert if not."

The Three Levels of Task Notification

Level 1: Completion Notification

The minimum: tell Boss when a long task is done. This is already implemented via notify-boss.js and run-claude.js. Every task that takes >1 minute must call this on completion. Non-negotiable rule in AGENTS.md.

Level 2: Progress Updates for Very Long Tasks

For tasks that take 5+ minutes, silence is uncomfortable even with a completion promise. The pattern: send a "started" notification with an estimated time, then a "done" notification at the end. For 10+ minute tasks, add a midpoint heartbeat: "still working, 50% done, ~5 more minutes."

This is the difference between waiting for an Uber with a real-time map and waiting for a cab you called 20 minutes ago and aren't sure is coming.

Level 3: Failure and Recovery Notification

The hardest case: a task fails mid-way. Jarvis runs out of context, a subagent crashes, an API returns an error, or the task encounters an edge case it can't handle. Without an explicit failure notification, Boss only discovers the failure when he checks in — which could be hours later.

The solution: wrap every long task in a try/catch equivalent with automatic failure notification including:

  • What the task was
  • What stage it failed at
  • What the error was (in plain English)
  • What Boss needs to do, if anything

Tools for Task Tracking Beyond Notifications

Mission Control Dashboard (existing)

The MC dashboard at localhost:3333 is already the best real-time view of what Jarvis is doing. It shows active agents, their status, scheduled jobs, and recent events. For Boss Henry, this is the "at a glance" view.

Current gap: the dashboard shows agents and jobs, but not fine-grained task progress within a single agent run. A blog post series like this one shows up as one "running" agent, not "post 6 of 10 complete."

Temporal — Production Workflow Orchestration

GitHub: github.com/temporalio/temporal · Stars: 12,000+

Temporal is the production-grade solution to long-task tracking at scale. It's used by Stripe, Netflix, and Uber for orchestrating complex workflows that run for minutes, hours, or even days. Features:

  • Durable execution: If your process crashes midway, Temporal automatically resumes from where it left off
  • Workflow visibility: Every task has a full execution history — what happened, when, what the results were
  • Timeouts and retries: Define exactly how long a step should take, how many retries before failure, and what to do on timeout
  • Human-in-the-loop: Workflows can pause and wait for human input, then resume automatically

For OpenClaw: Temporal is the long-term architecture for managing complex agent workflows. Not needed today, but understanding it shapes how you design the current system. The key pattern to borrow: every task has explicit states (queued, running, waiting, completed, failed), and state transitions trigger notifications.

BullMQ — Task Queues for Node.js

GitHub: github.com/taskforcesh/bullmq · Stars: 6,000+

BullMQ is a battle-tested Redis-based queue system for Node.js. If OpenClaw ever needs a proper task queue — where tasks are added, processed by workers, and tracked — BullMQ is the right choice. It has built-in retry logic, rate limiting, delay scheduling, and event callbacks for task state changes.

For Jarvis: a BullMQ queue could replace the current combination of active-tasks.json + HEARTBEAT.md for managing ongoing tasks. Tasks enter the queue, workers (subagents) process them, results are stored, notifications fire on state changes.

Designing Good Notification Messages

A bad completion notification: "Done." (What's done? Is it working? What do I do next?)

A good completion notification:

  • ✅ What was done: "10 blog posts written and pushed to henrys-blog"
  • 🔍 What was verified: "Git push succeeded, posts visible at /posts/ directory"
  • 🌐 Where to check: "Live at hungpham789.github.io/henrys-blog/"
  • ⚠️ Any caveats: "2 posts had placeholder images — update manually"
  • ⏱️ Elapsed time: "Took 28 minutes total"

This format gives Boss everything needed to decide "great, moving on" or "need to follow up on X" — in a single glance without opening any other tool.

The Silence Problem

The worst pattern: Jarvis starts a task, goes silent for 20 minutes, then either finishes or fails — and Boss doesn't know which.

The AGENTS.md rule is clear: "Boss should NEVER have to ask 'are you done?'" This is a design principle, not just a courtesy. Pushing completion state removes the cognitive overhead of tracking what Jarvis is doing, which is exactly the point of having an agent.

Practical rule: Any task that takes >1 minute needs a notify-boss.js call at the end. This is already in AGENTS.md as non-negotiable. For tasks that take >5 minutes, add a "started" notification too. For tasks that take >10 minutes, check in at the midpoint. This 3-tier system covers 99% of cases without notification fatigue.

✅ YES — System is Good, Polish It

Verdict: Notification infrastructure is solid — make failure notifications explicit

The completion notification system with notify-boss.js is working well. The gap is failure and timeout notifications — tasks that fail silently or take much longer than expected.

Do this week: Add a try/catch wrapper in run-claude.js that sends a failure notification with error details if a subagent exits with an error code.

Consider later: BullMQ for task queue management if/when the number of concurrent Jarvis tasks increases beyond what active-tasks.json handles cleanly. Temporal for production Antera workflows.
