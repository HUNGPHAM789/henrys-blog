Building a Route Inspector for Your Next.js App — in One Claude Code Run

March 25, 2026 6 min read AI, Next.js, Developer Tools
Back to all posts

When your Next.js app hits 30+ routes, you lose track of what's where. Which pages are protected? Which components does /teacher/dashboard import? What links go where? I had this problem with Antera — our ESL platform with 33 routes across teacher, student, and public flows.

So I asked Jarvis (my AI assistant running on OpenClaw) to build a tool that solves it. One prompt. One Claude Code run. Done in under an hour.

The Problem

Antera has routes like /teacher/generator, /student/drills/[id], /join/[code] — spread across nested folders. When I want to understand how the app flows, I'm constantly jumping between files in VS Code, tracing imports, trying to remember if a route is server or client rendered.

What I actually wanted was something like a living sitemap — something I could open in a browser, click a route, and immediately see: the source code, the auth protection status, the components it imports, and what other routes it links to.

💡
The insight: Your Next.js app/ folder is already a perfect data source. Every page.tsx is a node. Imports are edges. Auth middleware is metadata. You don't need a database — just parse the filesystem.

What We Built

The Antera Route Inspector is a local web app + MCP server, two things in one package:

🗺️

Route Tree

All 33 routes grouped by Teacher / Student / Public, with auth and layout info.

📄

Route Detail

Source code with syntax highlighting, imported components, data fetching, and outbound links.

🔗

Flow Map

Visual graph of route connections, color-coded by type (teacher/student/public/API).

🤖

MCP Server

6 tools for Claude Code agents to query the codebase without reading files manually.

How It Works

The server is a simple Express app that walks the src/app/ directory on startup, finds every page.tsx, parses it with a lightweight AST scanner, and caches the results in memory. File watching via chokidar invalidates the cache when files change.

No build step. No config. Just:

cd C:\antera-inspector
node server.js
# → http://localhost:3004

The MCP server runs separately as a stdio process, registered in ~/.claude/mcp.json. Claude Code agents can call it directly to explore the codebase without doing their own file traversal.

The MCP Tools

Six tools, all returning structured JSON:

get_all_routes()

All routes with type, layout, and auth status.

get_route_detail(route)

Full source, links, components, layout for one route.

find_component_usages(name)

All routes importing a given component.

get_navigation_map()

Full navigation graph as nodes + edges.

search_code(query)

Grep across all .tsx/.ts files in the project.

get_page_props(route)

Extract server component data fetching info.

Why This Matters for AI Agents

When you give Claude Code a task like "add a notifications bell to the teacher dashboard", it normally has to read a dozen files to understand where the nav lives, what components exist, and what patterns are already in use. With the MCP server, it can call get_route_detail('/teacher/dashboard') and get everything in one shot.

🤖
The meta-trick: We built a tool that makes the AI better at building more tools. The Route Inspector is itself used by AI agents to build features in the app it inspects.

How to Build One for Your App

1
Write the spec

Describe what you want: "A local web app that lists all my Next.js routes, shows source code, imports, and navigation links. Also expose this as an MCP server for Claude Code."

2
Spawn Claude Code with a task file

Write the spec to a task.md file and run claude --permission-mode bypassPermissions --print "$(cat task.md)" in an empty directory.

3
Register the MCP server

Add it to ~/.claude/mcp.json and use "type": "stdio" with "command": "node" pointing to your server file.

4
Iterate

Open the UI, find what's missing, and ask Claude to add it. The whole thing is vanilla JS + Express — easy to extend.

The Build Prompt

For reference, here's roughly what we gave Claude Code (condensed):

Build a local web tool called "Antera Route Inspector" at C:\antera-inspector.

Requirements:
- Express server on port 3004
- Scan C:\eslp-app\src\app for all page.tsx files
- Parse each file: extract route path, auth status, imported components,
  navigation links, data fetching patterns
- Web UI: sidebar route tree + detail panel + flow map
- MCP server (stdio, JSON-RPC 2.0) with 6 tools:
  get_all_routes, get_route_detail, find_component_usages,
  get_navigation_map, search_code, get_page_props
- File watching with cache invalidation
- VS Code integration (open file links)
✅
Result: Claude Code built the full app in one run — server, UI, MCP server, file watching, syntax highlighting. Total time: under an hour including testing. Zero manual code written.
